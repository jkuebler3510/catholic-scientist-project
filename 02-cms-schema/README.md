# 02 — CMS Schema (Sanity)

## Mission

Define every Sanity content type, the editorial workflow, the live-preview wiring, and the on-demand revalidation webhook so editors can run the site after launch with zero engineering involvement.

## Why this matters

Sanity is the system of record for editorial content. If the schema is right, every consuming page is a thin GROQ query. If the schema is wrong, downstream agents will hand-roll workarounds and the site will fragment.

## Inputs (read these first, in order)

1. `/README.md`
2. `/ARCHITECTURE.md` — "Content layer" and "Data contracts" sections
3. `/INFORMATION_ARCHITECTURE.md` — full document
4. `content-models.md`
5. `editorial-workflow.md`
6. `preview-and-revalidation.md`
7. `groq-queries.md`

## Deliverables

- `apps/studio/sanity.config.ts` with the project ID and dataset configuration.
- `apps/studio/schemas/` containing every content type listed in `content-models.md`.
- `apps/studio/desk-structure.ts` shaping the editorial UI (singletons pinned, content grouped).
- `apps/studio/plugins/` if needed (Portable Text customizations, scheduled publishing).
- `apps/web/lib/sanity/client.ts` — the read client (with CDN + token fallback for drafts).
- `apps/web/lib/sanity/queries.ts` — GROQ queries used across the site.
- `apps/web/lib/sanity/types.ts` — generated by `sanity-codegen` or hand-written; downstream folders import from here.
- `apps/web/app/api/webhooks/sanity/route.ts` — revalidation webhook (signature verified).
- `apps/web/app/(preview)/` route group with the preview drafts mode.
- Seed data: a sample document for every type so downstream pages have something to render.
- Documentation in this folder finalized to match what was actually built.

## Acceptance criteria

- [ ] `pnpm --filter studio dev` boots the Studio with all schemas registered, no schema errors.
- [ ] Every schema in `content-models.md` is implemented with the documented fields.
- [ ] Singleton documents (`siteSettings`, `homepage`) cannot be duplicated.
- [ ] Portable Text editor includes the configured marks, blocks, and embeds.
- [ ] Preview mode works end-to-end: editor clicks "Preview" → the public app shows the draft.
- [ ] Revalidation webhook verifies signature and revalidates the right tags on publish.
- [ ] Generated types are exported and consumable by `apps/web` with no `any`.
- [ ] Seed data makes every page in `INFORMATION_ARCHITECTURE.md` renderable.

## Out of scope

- Page-level rendering (lives in `03`–`06`).
- Content migration from the existing site (lives in `12-content-migration`).
- Search infrastructure (lives in `09-search-discovery`).
- Auth / member fields — those are Postgres, not Sanity (lives in `07-membership`).

## Open questions

- Do we want **scheduled publishing** in v1? Recommendation: yes (use `@sanity/scheduled-publishing`).
- Do editors need **multi-language**? No — see `TECH_DECISIONS.md`.
- How are images served — Sanity CDN via `@sanity/image-url`, or proxied through `next/image`? Recommendation: use Sanity CDN URLs as the source for `next/image`'s `loader` so we get both Sanity transforms and Next.js optimization. Document in `groq-queries.md`.
