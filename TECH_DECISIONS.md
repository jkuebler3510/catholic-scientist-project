# Technology Decisions

Every choice here is a decision, not a placeholder. Each entry explains the rationale, what was considered, and the consequences agents need to know about.

## Framework: Next.js 14+ (App Router) + TypeScript

**Why.** Best-in-class React framework, native SSR/SSG/ISR, server actions remove boilerplate APIs, App Router's React Server Components fit a content site perfectly (most pages ship near-zero JS), and Vercel deployment is friction-free.

**Considered.** Astro (would have been excellent for the content side, but loses ground when we add the member portal — too many client islands of authenticated UI). Remix (great DX, but smaller ecosystem and Vercel parity is not as tight). Plain WordPress theme (rejected — full rebuild was the user's stated goal).

**Consequences.** Next.js App Router is the assumed runtime. Use Server Components by default; mark `'use client'` only when needed. Use Server Actions for mutations rather than route handlers when called from the same app.

## Styling: Tailwind CSS + shadcn/ui

**Why.** Tailwind is the de facto standard for Next.js codebases — utility-first lets the design system stay token-driven, and `tailwind.config.ts` is the natural home for the brand tokens. shadcn/ui gives us accessible, unstyled-by-default Radix primitives we can theme to match SCS branding without inheriting another design system's opinions.

**Considered.** CSS Modules (more verbose, no co-located design tokens). Vanilla Extract (excellent type safety but unfamiliar to most contributors). Chakra/MUI (too opinionated visually).

**Consequences.** All design tokens live in `tailwind.config.ts` (and a CSS-variable theme layer for dark mode if we add it later). Never write inline `style={{}}`. Use `cn()` (clsx + tailwind-merge) for conditional classes.

## Content: Sanity (headless CMS)

**Why.** Real-time collaborative editing, structured content with Portable Text, generous free tier for non-profits, excellent Next.js integration including a live preview API and on-demand revalidation webhooks. The Studio is React and lives in our repo, so schema and code evolve together.

**Considered.** Contentful (more expensive, less flexible Portable Text equivalent). Payload (great, but self-hosted operational burden). Decap CMS / TinaCMS (Git-based — would have been fine for a smaller content team, but SCS publishes events frequently and a real CMS is worth it). Strapi (self-hosted, ops burden again).

**Consequences.** All editorial content is fetched via GROQ from Sanity. Types are generated by `sanity-codegen`. Editors do not touch the codebase.

## Auth: Clerk

**Why.** Drop-in UI, Next.js-first SDK, built-in MFA, magic links, organizations (useful for chapter administrators), and a generous free tier. We do not want to build auth ourselves and we don't need the configurability of Auth.js.

**Considered.** Auth.js / NextAuth (free forever, but requires us to build the email flow, the management UI, and the verification logic — months of work). Supabase Auth (good, but pulls in Supabase as a database choice, which we don't otherwise need). Clerk's per-user pricing is acceptable at SCS scale.

**Consequences.** Clerk owns the user record. Our Postgres `members` table joins by Clerk `userId`. Use the Clerk middleware for route protection and the `<SignIn>` / `<UserProfile>` components in the portal.

## Payments: Stripe (Checkout + Customer Portal)

**Why.** Industry standard, excellent docs, supports one-time and recurring, the Customer Portal gives members self-service for free, and tax compliance (Stripe Tax) is one click away when needed.

**Considered.** PayPal alone — rejected. Donations-specific platforms (DonorBox, Givebutter) — they take a percentage and lock the donor data into their system.

**Consequences.** Use Stripe Checkout (hosted) for the donation and dues flows, not custom forms. Use the Customer Portal for ongoing management. Webhooks must be implemented and signature-verified.

## Database: Postgres (Neon)

**Why.** Battle-tested, perfect fit for relational application data (members, RSVPs, dues ledger). Neon offers branching environments that mirror our preview/prod model, generous free tier, and serverless-friendly connection pooling.

**Considered.** Supabase Postgres (excellent — the only reason we picked Neon is we already chose Clerk for auth, so Supabase's auth is wasted). PlanetScale MySQL (no native JSON & arrays, less Postgres-y). SQLite (won't work on serverless Vercel).

**Consequences.** ORM choice is open (Drizzle vs. Prisma) — `00-foundation` decides. All schemas live in `lib/db/schema.ts`. Migrations run in CI before deploy.

## Hosting: Vercel

**Why.** Best Next.js experience, free preview deploys per PR, edge functions, automatic image optimization, integrated logs and analytics. Aligns with the entire stack.

**Considered.** Cloudflare Pages (Next.js support is improving but not at parity for App Router). Self-host on AWS (operational burden not justified at SCS scale).

**Consequences.** A few Vercel-specific features are acceptable: ISR, on-demand revalidation, image optimization. Do not adopt Vercel KV or Edge Config without a comparable self-hostable fallback noted in `00-foundation`.

## Forms & validation: React Hook Form + Zod

**Why.** Smallest, fastest form library; Zod provides one schema for client-side validation, Server Action argument parsing, and CMS-side type narrowing.

**Consequences.** Every form has a Zod schema. Server Actions parse with the schema before any business logic.

## Email: Resend + React Email

**Why.** Resend is purpose-built for transactional email from Next.js, React Email lets us write emails in JSX, both have free tiers.

**Consequences.** All transactional email (welcome, dues receipt, RSVP confirmation, application acknowledgment) is templated in `packages/emails`. Marketing email (newsletter blasts) is **out of scope for v1** — see `04-news/` for the deferred plan to integrate Buttondown or Beehiiv.

## Search: Sanity GROQ (v1)

**Why.** Already have it, free, fast enough for the content volume on launch day.

**Consequences.** Wrap the search behind an interface in `09-search-discovery` so a future swap to Algolia or Typesense doesn't require touching every page.

## Video: YouTube embeds (v1)

**Why.** SCS already publishes lectures to YouTube. Embedding via `lite-youtube-embed` keeps the bundle small and respects performance.

**Considered.** Self-hosting on Mux or Cloudflare Stream — overkill for v1 traffic.

**Consequences.** A `talk` document in Sanity stores the YouTube ID. The component renders a lite embed.

## Maps: MapLibre + protomaps tiles (v1)

**Why.** For the chapters map. Free, self-hostable tiles, no Mapbox token required.

**Consequences.** Lazy-loaded, behind a "show map" interaction so we don't pay the JS cost on first load.

## Internationalization: deferred

**Why.** SCS is global but currently English-only. Architect to allow `next-intl` later (no hard-coded URLs that conflict with `/[locale]` routing).

## Testing

- **Unit.** Vitest, co-located with source.
- **Component.** React Testing Library + Vitest.
- **E2E.** Playwright for critical flows (apply → pay dues → see member portal; donate; RSVP).
- **Visual regression.** Optional. If adopted, use Chromatic against Storybook for design-system components only.

## CI/CD

- GitHub Actions runs typecheck, lint, unit tests, and Playwright on every PR.
- Vercel handles deploys (preview per PR, prod on `main`).
- Sanity deploys via `sanity deploy` from a separate workflow when `studio/` changes.

## Things explicitly *not* in v1

- Forum / discussion software (the existing "Ideas & Discussions" is one-way published essays, not a forum).
- Newsletter sending (we'll embed a signup form; sending is out of scope until a tool is chosen post-launch).
- AI features (search rewrite, summaries, etc.).
- Internationalization.
- Native mobile app.
- An admin dashboard beyond what Sanity Studio + Stripe Dashboard + Clerk Dashboard + a thin custom Postgres view provides.
